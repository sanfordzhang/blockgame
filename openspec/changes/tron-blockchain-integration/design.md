# Technical Design: TRON Blockchain Integration

## Context

当前系统是一个基于 React + Node.js 的德州扑克游戏，使用虚拟筹码（内存存储）：
- 前端：React + Socket.IO Client
- 后端：Express + Socket.IO Server
- 存储：内存（无持久化）
- 钱包：MetaMask/ethers（未实际使用）

需要改造为 TRON 区块链上的真实资金托管游戏。

### 约束条件

1. **TRX 原生限制**：TRX 没有 approve/transferFrom 机制（仅 TRC20 支持），必须采用托管模式
2. **性能要求**：游戏逻辑必须保持毫秒级响应，不能每次操作都上链
3. **Tron 出块时间**：约 3 秒，不适合实时游戏动作上链

## Goals / Non-Goals

**Goals:**
- 实现真实 TRX 资金托管与结算
- 支持娱乐模式（测试网）和真实模式（主网）
- 动态抽水率管理
- 保持现有游戏体验（混合架构：链下游戏 + 链上结算）

**Non-Goals:**
- 二期 BRIDGE 代币经济（后续迭代）
- 地区限制功能（目前不需要）
- KYC/AML 合规（后续迭代）
- VRF 随机数（一期使用简化方案）

## Decisions

### D1: 混合架构（链下游戏 + 链上结算）

**选择**: 混合架构
**原因**: 
- 纯链上方案不可行：每局约 20 个动作 × 3 秒出块 = 60+ 秒
- 混合方案：游戏逻辑保持链下实时执行，仅结算时上链一次

**架构图**:
```
┌─────────────────────────────────────────────────────────────┐
│                      混合架构                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  链下 (Socket.IO)          │        链上 (智能合约)          │
│  ─────────────────         │        ─────────────────       │
│  • 实时游戏逻辑             │        • 资金托管              │
│  • 毫秒级响应               │        • 结算验证              │
│  • Fold/Call/Raise         │        • 抽水收取              │
│                            │        • 余额管理              │
│                            │                                │
│  结算时机: 游戏结束后一次性上链                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**替代方案**:
- 纯链上：用户体验极差，不可行
- 状态通道：复杂度高，一期不采用

### D2: 托管模式（Custody Model）

**选择**: 用户先存入 TRX 到合约，游戏从合约余额扣除
**原因**: TRX 原生代币不支持授权机制，无法实现"锁定但不转移"

**流程**:
```
用户钱包 ──deposit──▶ 合约余额 ──joinTable──▶ 锁定余额
                                          │
                        ◀──withdraw──     │
                                        游戏
                                          │
                                          ▼
                     赢家 ◀──settleGame── 结算
```

**替代方案**:
- 授权模式：TRX 不支持
- 每局转账：Gas 费过高，用户体验差

### D3: 动态抽水率设计

**选择**: 合约状态变量 + 管理员可调 + 安全世界限制
**原因**: 运营需要根据市场情况调整抽水率

**参数设计**:
```
最小抽水率: 1% (100 basis points)
最大抽水率: 10% (1000 basis points)
调整频率限制: 每 24 小时最多调整一次（时间锁）
单次调整幅度: 最多 ±2%
```

**替代方案**:
- 固定抽水率：缺乏运营灵活性
- 无限制调整：存在恶意调整风险

### D4: 双网络部署（测试网 + 主网）

**选择**: 同一套合约代码，部署到两个网络
**原因**: 支持娱乐模式（测试网 TRX）和真实模式（主网 TRX）

**网络映射**:
| 模式 | 网络 | 合约地址 | 资金 |
|------|------|----------|------|
| 娱乐模式 | Nile Testnet | `TEST_CONTRACT_ADDR` | 测试网 TRX（免费） |
| 真实模式 | Tron Mainnet | `MAIN_CONTRACT_ADDR` | 真实 TRX |

**实现**:
- 前端根据用户选择的模式切换 TronWeb 实例
- 后端根据模式调用不同合约

### D5: 后端服务架构

**选择**: 新增区块链服务层，保持现有游戏服务不变
**原因**: 最小化改动，隔离关注点

**新增目录结构**:
```
server/
├── blockchain/                    # 新增
│   ├── index.js                   # 入口
│   ├── TronService.js             # Tron链交互
│   ├── ContractService.js         # 合约调用封装
│   └── EventListener.js           # 链上事件监听
├── services/
│   ├── GameSettlementService.js   # 新增：游戏结算
│   ├── AdminService.js            # 新增：管理服务
│   └── ...
└── models/
    ├── Player.js                  # 新增：玩家数据模型
    └── Transaction.js             # 新增：交易记录模型
```

## Risks / Trade-offs

### R1: 服务端作恶风险
**风险**: 服务端可能篡改游戏结果
**缓解**: 
- 实现 commit-reveal 模式
- 游戏过程哈希上链
- 用户可验证结果
- 开源代码 + 审计

### R2: 智能合约漏洞
**风险**: 合约漏洞可能导致资金损失
**缓解**:
- 使用 OpenZeppelin 安全库
- 多层审计（AI + 专业公司）
- 资金限额（初期限制单笔最大买入）
- 紧急暂停机制

### R3: 网络拥堵
**风险**: TRON 网络拥堵可能导致结算延迟
**缓解**:
- 交易队列管理
- 用户友好的状态提示
- 后端自动重试机制

### R4: 私钥安全
**风险**: 后端操作员私钥泄露
**缓解**:
- 环境变量存储，不提交代码库
- 考虑二期升级为多签钱包
- 限制操作员权限

## Migration Plan

### Phase 1: 测试网部署（第 1-4 周）

1. 搭建 Tron 开发环境
2. 编写并部署 BridgeGameV1.sol 到 Nile 测试网
3. 前端 TronLink 集成
4. 后端区块链服务开发
5. 娱乐模式全流程测试

### Phase 2: 主网部署（第 5-8 周）

1. 智能合约审计
2. 部署主网合约
3. 灰度发布（限制玩家数量）
4. 监控告警配置
5. 正式上线

### 回滚策略

1. 合约暂停：调用 `pause()` 停止所有操作
2. 资金撤离：调用 `withdrawContractBalance()` 提取合约余额
3. 前端回退：切换回虚拟筹码模式（保留代码分支）

## Open Questions

1. **抽水率时间锁具体时长**：建议 24 小时，是否需要调整？
2. **单笔最大买入限额**：建议 1000 TRX，是否合理？
3. **管理后台是否需要独立域名**：还是集成在主站（权限控制）？
4. **测试网 TRX 获取方式**：使用官方 Faucet 还是内置领取？
